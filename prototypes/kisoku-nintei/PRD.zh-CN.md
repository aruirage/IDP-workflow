# NeosAI 日本保险 IDP 工作流配置 — 产品需求文档（PRD）

## 第 6 章｜功能详细说明

### 6.01 业务场景设置

#### 6.01.1 概述

四步向导：Step1 案件集约规则，Step2 工作流画布，Step3 通知设置，Step4 输出字段与发布。新建、编辑已发布、复制场景共用流程。案件场景只展示一个主状态，不展示分步状态。各步骤保存只写草稿配置；只有发布生成正式版本号。

#### 6.01.2 功能点

1. 场景列表：新建、复制、重命名、删除、关键词检索；复制后进入 Step1 继续编辑。
2. Step1：配置名称、说明、账票集合（1～20）、主账票与主字段、关联账票字段关系、匹配策略；支持关联检查与 AI 自动关联。
3. Step2：进入工作流画布；画布读取 Step1 账票与关系，限制节点可选范围。工具栏提供工作流测试；测试结论不阻断进入 Step3/Step4，但测试通过是发布条件。
4. Step3：配置全局通知规则模板。每条规则选择対象节点、触发事件、通知对象、件名、内容和可插入变量；页面不展示触发表达式。处理节点按节点专属 `Status / Result` 自动判断；人工确认按完成、补件、案件终止三出口事件触发。
5. Step4：从 OCR 抽出账票类型中勾选要导出的字段（允许 0）。点保存固化勾选；0 字段也可以保存，表示用户确认本场景不导出 OCR 字段。
6. 发布：场景须为可发布，且 Step3、Step4 的草稿配置均已保存有效。发布时生成新版本号；新案件只读取已发布版本。

保存与版本号：每一步保存只保存草稿配置，不生成版本号。Step1 保存只保存场景基础与集约规则；Step2 保存只保存画布、节点、连线和变量引用；Step3 保存只保存通知配置；Step4 保存只保存输出配置。发布读取 Step1～Step4 当前已保存草稿，生成一个新的正式版本号。

每一步状态变更：


| 动作         | 保存内容             | 测试结论                     | 场景主状态                    | 说明                                  |
| ---------- | ---------------- | ------------------------ | ------------------------ | ----------------------------------- |
| 新建/复制场景    | 初始化草稿            | 未测试                      | 草稿                       | 不生成版本号                              |
| Step1 保存成功 | 场景基础、账票集合、关联规则   | 若 Step1 有实质变更则置为未测试      | 草稿                       | 不覆盖 Step2～Step4 已保存内容；若破坏后续引用则阻断保存  |
| Step2 保存成功 | 画布、节点、连线、变量引用    | 若画布有实质变更则置为未测试           | 草稿                       | 不阻断进入 Step3；未测试或测试失败时不可发布           |
| Step2 测试成功 | 不改变配置，只写测试结论     | 成功                       | 按 Step3 和 Step4 保存情况重算 | 只有 Step3 和 Step4 也已保存有效时，主状态才显示为可发布；否则仍为草稿 |
| Step2 测试失败 | 不改变配置，只写测试结论     | 失败                       | 草稿                       | 可继续进入 Step3/Step4 配置；公开按钮不可用        |
| Step3 保存成功 | 通知规则模板           | 不改变 Step2 测试结论           | 按 Step2 测试和 Step4 保存情况重算 | 通知规则为空也可保存；有规则时只校验必填字段和变量有效性        |
| Step4 保存成功 | 输出字段勾选           | 不改变 Step2 测试结论           | 按 Step2 测试和 Step3 保存情况重算 | 允许保存 0 个输出字段                        |
| 发布成功       | 读取四步当前已保存草稿并生成版本 | 保留成功结论                   | 已发布                      | 生成正式版本号；新案件只读取已发布版本                 |
| 已发布后编辑任一步  | 从已发布版本派生草稿       | 若 Step1/Step2 实质变更则置为未测试 | 草稿                       | 旧发布版本继续服务正式案件，直到再次发布                |


已发布后再编辑：打开已发布场景并修改任一步时，系统从已发布版本派生草稿编辑态。保存后重新打开，继续看到已保存的草稿改动，不需要重新改；正式案件仍使用上一个已发布版本。只有再次通过 Step2 测试、保存 Step3/Step4，并点击发布后，才生成新版本并替换后续新案件使用的版本。

#### 6.01.6 Step1 案件集约规则

Step1 负责业务场景基础信息、关联账票集合、主账票、主字段、账票间字段关系和匹配策略。Step2 的账票候选、OCR 字段候选、条件字段候选和 Step4 的输出候选都从 Step1 派生。

Step1 保存只写场景基础与集约规则，不生成版本号，不执行工作流测试。保存前检查必填项、账票数量上限和后续引用保护；通过后进入 Step2。

关联检查只检查一件事：是否存在没有和主账票连上的关联账票。账票数量、主账票、主字段、匹配策略在添加账票或字段选择时已经校验；关系字段有效性、关系方向、重复关系和后续引用保护不属于关联检查。


| 规则 ID | 检查项       | 通过条件                 | 失败条件 / 处理               |
| ----- | --------- | -------------------- | ----------------------- |
| S1-01 | 关联账票连接主账票 | 每个关联账票至少有 1 条关系连到主账票 | 任一关联账票未连到主账票时，提示未关联账票列表 |




#### 6.01.7 Step2 工作流画布

Step2 负责配置工作流节点、连线、条件表达式、人工确认出口和节点参数。Step2 读取 Step1 的账票集合、主账票、关联关系和 OCR 字段；不能在 Step2 单独新增、删除或重命名账票类别和 OCR 字段。

Step2 提供工作流测试，测试结论是发布门禁，不是进入 Step3 的门禁。画布测试弹窗使用内置「标准集约完成案件」样例，只读；点「テスト実行」后检查当前画布配置是否能跑通。测试未通过也允许进入 Step3/Step4 继续配置，但场景保持草稿且不可发布；画布、节点配置、连线或变量引用发生实质变更后，测试结论作废。


| 检查维度 | 检查内容                                                  | 失败处理                     |
| ---- | ----------------------------------------------------- | ------------------------ |
| 结构   | 开始、结束、有效路径、孤立节点、回流可终止                                 | 测试失败；高亮问题节点或断点；保持草稿且不可发布 |
| 连线   | 输入数量、条件出口、人工确认出口、悬空连线                                 | 测试失败；提示对应连线或出口问题         |
| 节点配置 | 前处理、OCR 抽出、数据映射、AI 检证、人工确认、自定义函数                      | 测试失败；提示缺失配置或非法依赖         |
| 变量引用 | 条件变量、自定义函数输入、删除或改线后的引用                                | 测试失败；提示失效变量和节点           |
| 字段选择 | 标准字段和 OCR 字段必须选到叶子字段；禁用 Array、Date、DateTime、`files[]` | 测试失败；提示重新选择字段            |
| 模拟执行 | 按画布路径生成时间线；人工确认跳过真实等待                                 | 任一节点失败即停止本轮测试            |




#### 6.01.8 Step3 通知设置

Step3 负责配置全局通知规则模板，不读取 Step2 画布里的实际节点实例。运行时由工作流节点类型和事件匹配这里的全局规则；如果 Step2 没有对应节点，则对应规则不会被触发，但配置本身仍可保存。

页面上方是「通知ルール追加」生成器。配置人员先选対象ノード，再选トリガーイベント，然后选择通知对象，编辑件名和内容。件名和内容上方提供变量插入区，先切换插入目标（件名 / 内容），再从变量下拉中选择要插入的变量。

页面下方是「通知ルール一覧」。每条规则展示 No.、対象ノード、トリガーイベント、通知对象、件名、内容摘要、状态和操作。每条规则可开启、关闭、编辑、删除。点击「編集」打开弹窗编辑器，不回填到页面上方生成器。


| 配置项      | 控件   | 必填  | 规则                                       |
| -------- | ---- | --- | ---------------------------------------- |
| 対象ノード    | 单选下拉 | 是   | 固定为前处理、OCR 抽出、数据映射、AI 检证、人工确认            |
| トリガーイベント | 单选下拉 | 是   | 随対象ノード切换；处理节点为处理失败、结果要确认；人工确认为完成、补件、案件终止 |
| 通知对象     | 多选下拉 | 是   | 可多选；允许同一対象ノード、同一トリガーイベント、同一通知对象保存多条规则    |
| 件名       | 文本   | 是   | 支持插入变量                                   |
| 内容       | 多行文本 | 是   | 支持插入变量                                   |
| 状态       | 开关   | 是   | 开启规则参与运行时匹配；关闭规则保留配置但不触发                 |


通知可插入变量：


| 变量 | 插入格式 | 展示条件 | 说明 |
| ---- | ---- | ---- | ---- |
| 案件 ID | `{caseId}` | 全事件 | 通知对应案件的唯一标识 |
| 触发节点状态 | `{status}` | トリガーイベント = 处理失败 | 运行时写入触发节点的 status；用于说明失败状态 |
| 触发节点结果 | `{result}` | トリガーイベント = 结果要确认 | 运行时写入触发节点的 result；用于说明要确认的业务结果 |
| 人工确认分支 | `{branch}` | トリガーイベント = 完成 / 补件 / 案件终止 | 运行时写入人工确认选择的出口分支 |


通知变量选择器规则：

1. Step3 通知变量不是 Step2 出力変数池，不读取具体画布节点的前序输出。
2. 变量选择器只展示 `{caseId}`，以及由当前トリガーイベント决定的 `{status}` 或 `{result}`。
3. 处理失败事件只展示 `{caseId}` 和 `{status}`；结果要确认事件只展示 `{caseId}` 和 `{result}`；人工确认的完成、补件、案件终止事件只展示 `{caseId}` 和 `{branch}`。
4. 变量插入只影响件名和内容，不允许用户配置触发表达式。

内置触发事件：


| 通知事件  | 可配置节点                 | 触发条件                                                                     | 说明                  |
| ----- | --------------------- | ------------------------------------------------------------------------ | ------------------- |
| 处理失败  | 前处理、OCR 抽出、数据映射、AI 检证 | `{node}.case.*Status = failed`                                           | 代表系统、依赖或配置异常        |
| 结果要确认 | 前处理、OCR 抽出、数据映射、AI 检证 | `{node}.case.*Status = success AND {node}.case.*Result = reviewRequired` | 代表节点正常跑完，但业务结果需要人工看 |
| 完成    | 人工确认                  | 人工确认节点选择完成出口时触发                                                          | 与完成出口路由同时发生         |
| 补件    | 人工确认                  | 人工确认节点选择补件出口时触发                                                          | 与补件待办或补件流程同时发生      |
| 案件终止  | 人工确认                  | 人工确认节点选择案件终止出口时触发                                                        | 与案件终止路由同时发生         |


校验规则：


| 规则 ID | 检查项      | 通过条件                                              | 失败条件 / 处理                           |
| ----- | -------- | ------------------------------------------------- | ----------------------------------- |
| S3-01 | 进入 Step3 | Step2 当前配置可保存                                     | 测试未完成或失败不阻断进入；仅在 Step3 顶部提示当前场景不可发布 |
| S3-02 | 可配置节点    | 対象ノード在固定列表内                                       | 不允许选择开始、结束、条件、自定义函数                 |
| S3-03 | 必填字段     | 启用规则必须填写対象ノード、トリガーイベント、通知对象、件名、内容                 | 缺任一必填项时提示补齐                         |
| S3-04 | 变量合法性    | 件名和内容只能使用当前トリガーイベント允许的变量；处理失败只允许 `{caseId}` / `{status}`，结果要确认只允许 `{caseId}` / `{result}`，人工确认出口只允许 `{caseId}` / `{branch}` | 出现不允许变量时阻断保存并提示变量名 |
| S3-05 | 内置触发     | 页面不展示触发表达式；处理节点按 `Status / Result` 判断；人工确认按出口事件判断 | 不允许用户改写触发条件                         |

| 动作    | 规则                                                       |

| 保存    | 通知规则为空时允许保存；有规则时只执行必填字段和变量有效性校验                          |

#### 6.01.9 Step4 输出字段与发布

Step4 负责配置导出字段和发布。侧栏展示 Step2 OCR 抽出已开启的账票类型；右侧表格按字段行展示，只有字段勾选和排序是配置项；OCR 抽出列和照合结果列都是样例预览，用于帮助配置人员判断字段含义，不作为独立导出配置。允许 0 个字段，表示用户确认本场景不导出 OCR 字段。

Step4 没有测试。点击「保存」固化当前勾选并记为已保存；保存不改写 Step1～Step3，也不重验工作流。公开时只检查场景主状态是否为可发布；发布成功后生成新版本号并标记已发布。


| 规则 ID | 检查项      | 通过条件                                           | 失败条件 / 处理                                |
| ----- | -------- | ---------------------------------------------- | ---------------------------------------- |
| S4-01 | 进入 Step4 | Step3 通知配置保存有效                                 | Step3 保存校验失败时阻断进入                        |
| S4-02 | 候选来源     | 侧栏仅展示 Step2 OCR 抽出已开启账票类型；右侧表格列为字段、OCR 抽出样例、照合结果样例；允许 0 勾选 | 无 OCR 候选时整页空态；不因此阻断保存                    |
| S4-03 | 保存       | 用户点保存即固化当前勾选并记为已保存                             | 保存失败时保持未保存                               |
| S4-04 | 公开       | 主状态为可发布                                        | 未满足时提示缺失的测试或保存项；输出字段允许 0 个，但 Step4 必须保存过 |




#### 6.01.10 步骤跳转、引用保护与联动

Step1 提供关联检查；Step2 提供工作流测试，并作为发布门禁。Step3 没有测试，只保存通知配置；Step4 没有测试，只保存输出勾选。Step2 测试未通过也可以进入 Step3/Step4 继续配置，但不能发布。终了结构与到达在 Step2 测试内检查。公开时：须处于可发布主状态。

### 6.02 工作流画布与可编排节点



#### 6.02.1 概述

画布只编排**案件集约完成之后**的处理段；不编排上传、案件集约、导出。节点规则在账票类型、数据映射、AI 检证等设置页维护，画布配置外壳、开关、引用与分支。

- 各节点变量统一说明出力、消费路径、允许取值和可用运算符。
- 配置端「出力変数」不提供折叠按钮；列表行只显示裸变量键 + 类型；点圆形 i 查看变量显示名和取值范围。i 气泡标签用日文（変数名 / 取値範囲 / 例），不展示长说明；`files[]` 只附 1 条短 JSON 示例。所有节点都直接平铺展示，不再显示案件/文件/账票类型分组标题。
- 画布节点不再展示配置摘要行；节点卡片只显示类型名（条件/人工确认额外显示分支结构）。



#### 6.02.2 功能点

1. 画布编辑：添加/插入/连接/删除节点与连线、自动整列；逻辑变更置测试为未测试。节点标题固定为类型名。
2. 节点选择器：业务 Agent（前处理、OCR、映射、检证）、控制（条件、人工确认）、扩展（自定义函数）、端点（开始、结束）。
3. 节点编辑面板：按类型展示参数；底部「出力変数」不折叠，只读展示变量键 + 类型；变量显示名、取值范围、`files[]` 短示例在 i 中。映射、检证仅跳转设置页。
4. 变量引用：条件仅选可达上游变量；自定义函数通过「入力変数」追加入参，每个入参绑定开始节点 `files[]`、文件类别 JSON 变量或前序节点输出变量。会破坏既有引用的保存阻断，发布前仍发现不可达引用时发布阻断。
5. 分支：条件 IF/ELIF/ELSE；人工确认三出口（完成/补件/案件终止），审查者选出口即路由。
6. 人工确认：须接在前处理/OCR/检证之后；只接需要的出口，至少一条。
7. 工作流测试：内置集约完成用例；不跑上传/集约，不写正式案件/待办。测试成功后允许发布；测试失败不阻断进入 Step3/Step4，但阻断发布。
8. Step1 派生 schema：Step2 的账票类别和 OCR 字段只来自 Step1 关联账票集合；Step2 不能新增、删除或重命名账票类别和 OCR 字段。
9. 说明图标统一为圆形 i；画布不再提供「全部折叠节点」入口。复制场景后不展示 Step1「要確認」引用检查横幅。

#### 6.02.8 工作流测试

Step2 画布用工作流测试证明「当前画布配置可跑通」。测试通过是发布门禁，不是进入 Step3 的门禁；它不是正式跑件：不创建正式案件、不写真实待办。测试用例由系统内置提供，配置人员不能上传、编辑或替换用例。本节只定义测试用例策略、怎么操作、成功/失败写什么、弹什么提示。

功能点：

1. 配置人员在 Step2 打开测试弹窗，确认系统内置测试用例后执行测试。
2. 系统按「用例 → 结构 → 连线 → 节点配置 → 变量引用 → 字段选择 → 依赖 → 模拟执行 → 发布门禁」顺序校验；前一层阻断则不进入后一层。结束节点有无、无出边、可达结束等归结构/模拟维度，不在测试外再挂一层。
3. 模拟执行按样例字段求值后的命中路径生成时间线（条件走真分支，人工确认跳过真人动作后按已连接出口继续）；任一节点失败立即停止，后续节点保持未执行。
4. 全部规则通过：写测试结论=成功，场景状态置为可发布。
5. 任一失败（含未配置结束、不可达结束）：写测试结论=失败，场景保持草稿（非已发布时），阻断发布。
6. 人工确认节点在测试中跳过真实确认动作（不等人、不建真实待办），校验上下文与出口后继续下游，以便测通后续节点。正式运行仍须真人确认。

| 粒度      | 定义                              | 典型变量                                                                                                                                                  | 是否可进条件变量池              | 说明                    |
| ------- | ------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------- | --------------------- |
| 案件级变量   | 一个案件实例只有一个值，或由多个文件/字段汇聚成一个案件级结论 | `preprocessStatus`、`preprocessResult`、`ocrStatus`、`ocrResult`、`mappingStatus`、`mappingResult`、`verifyStatus`、`verifyResult`、`hitlStatus`、`codeStatus` | 只有消费分类含条件、且来自上游可达节点时可选 | 条件节点主要消费处理状态与处理结果     |
| 文件级变量   | 仅开始节点以 `files[]` 数组承载归属文件信息     | `files[]`                                                                                                                                             | 不可直接选                  | 传给处理节点与人工确认页读取；条件不用数组 |
| 账票字段级变量 | 从 Step1 账票模板字段或标准字段中选到具体叶子字段后生成 | `docTypes.{账票类型}.{OCR字段名}`、`standardFields.{字段名}`                                                                                                     | 当前版本按 String 类型可选      | 字段容器不可选；必须选到叶子字段      |


`files[]` 仅开始节点输出。人工确认页、日志和系统执行仍可读取运行时文件对象，但不作为其他节点的出力变量。

#### 处理状态与处理结果（业务处理节点共用）

前处理、OCR、数据映射、AI 检证统一拆成两个案件级枚举：


| 变量   | 取值               | 白话                               |
| ---- | ---------------- | -------------------------------- |
| 处理状态 | `processing`     | 还在跑                              |
| 处理状态 | `success`        | 跑完了且没系统失败；没东西可处理时也记成功（旧所说的 skip） |
| 处理状态 | `failed`         | 跑挂了（系统/配置/依赖异常）                  |
| 处理结果 | `passed`         | 业务通过，可走主流程                       |
| 处理结果 | `reviewRequired` | 业务要确认，去人工确认                      |


分岔时：先看处理状态有没有挂（`failed`），再看处理结果要不要人看（`reviewRequired`）。

Step3 通知规则模板只消费通知变量白名单里的变量。触发条件由系统按节点事件内置判断，用户只能在件名和内容里插入变量；插入变量不改变触发条件。

内置触发条件：


| 通知事件   | 可配置节点  | 触发条件                                                                                       | 说明                  |
| ------ | ------ | ------------------------------------------------------------------------------------------ | ------------------- |
| 处理失败   | 前处理    | `{node}.case.preprocessStatus = failed`                                                    | 代表系统、依赖或配置异常        |
| 处理失败   | OCR 抽出 | `{node}.case.ocrStatus = failed`                                                           | 代表系统、依赖或配置异常        |
| 处理失败   | 数据映射   | `{node}.case.mappingStatus = failed`                                                       | 代表系统、依赖或配置异常        |
| 处理失败   | AI 检证  | `{node}.case.verifyStatus = failed`                                                        | 代表系统、依赖或配置异常        |
| 结果要确认  | 前处理    | `{node}.case.preprocessStatus = success AND {node}.case.preprocessResult = reviewRequired` | 代表节点正常跑完，但业务结果需要人工看 |
| 结果要确认  | OCR 抽出 | `{node}.case.ocrStatus = success AND {node}.case.ocrResult = reviewRequired`               | 代表节点正常跑完，但业务结果需要人工看 |
| 结果要确认  | 数据映射   | `{node}.case.mappingStatus = success AND {node}.case.mappingResult = reviewRequired`       | 代表节点正常跑完，但业务结果需要人工看 |
| 结果要确认  | AI 检证  | `{node}.case.verifyStatus = success AND {node}.case.verifyResult = reviewRequired`         | 代表节点正常跑完，但业务结果需要人工看 |
| 完成通知   | 人工确认   | 人工确认节点选择「完成」出口时触发                                                                          | 审查者确认完成，与完成出口路由同时发生 |
| 补件通知   | 人工确认   | 人工确认节点选择「补件」出口时触发                                                                          | 与补件待办/补件流程同时触发      |
| 案件终止通知 | 人工确认   | 人工确认节点选择「案件终止」出口时触发                                                                        | 通知案件终止相关对象          |


人工确认节点不输出 `result` 或 `action` 变量；完成、补件、案件终止三类通知由人工确认出口路由事件触发，不进入条件变量池。自定义函数节点不是 Step3 通知设置的対象ノード。开始节点、结束节点、条件节点不配置通知。

自定义函数节点和人工确认节点只输出各自节点状态键：`hitlStatus` / `codeStatus`，取值同处理状态：`processing / success / failed`。它们不输出 `result`、`action` 或错误消息变量。

#### 变量消费分类

案件状态是平台固定聚合状态，不是 Step2 节点变量。条件节点不能直接选择案件状态。


| 消费分类 | 谁消费              | 可消费变量                                         | 不可消费变量                                                   | 说明               |
| ---- | ---------------- | --------------------------------------------- | -------------------------------------------------------- | ---------------- |
| 条件   | 条件判断节点           | 上游可达、消费分类含条件的变量；标准字段和 Step1 OCR 字段必须选到叶子      | 开始节点输出、结束节点输出、`files[]`、Array 容器、Date、DateTime、系统变量      | 只用于 IF / ELIF 分岔 |
| 通知   | Step3 通知规则模板     | 通知变量白名单内的案件、节点、标准字段、OCR 字段和文件基础信息变量           | 触发条件本身不可配置；不得插入 Array 容器、Date、DateTime、条件节点输出、自定义函数脚本返回值 | 只用于主题和内容插值       |
| 系统   | 引擎、日志、状态汇聚、Step4 | 开始 `caseId` / `files[]`、映射 `standardFields` 等 | 除非变量表另行标注，不提供给条件                                         | 可在出力変数面板展示       |
| 待办   | 人工确认页、待办列表、审计详情  | 运行时文件与确认结果等                                   | 条件表达式直接比较数组                                              | 分岔使用状态/结果变量      |


同一业务事件的推荐写法：


| 场景    | 条件变量                                     | 待办/运行时    |
| ----- | ---------------------------------------- | --------- |
| 结果要确认 | `{node}.case.*Result = reviewRequired`   | 人工确认页展示明细 |
| 节点失败  | `{node}.case.*Status = failed`           | 异常分支或终止   |
| 全部通过  | `*Status = success` 且 `*Result = passed` | 走主流程      |




#### 条件变量规则

条件节点只能选择上游连线可达变量。开始节点、结束节点、条件节点本身都不是条件变量来源。条件节点变量选择器不得出现 Array 容器、Date、DateTime、`files[]`、字段容器 `standardFields`。

账票类型字段不经由开始节点输出，条件选择器直接从 Step1 模板加载 `docTypes.{账票类型}.{OCR字段名}`。

条件运算子统一使用英文：


| 类型                      | 可用运算子                                                                 | 默认     | 说明              |
| ----------------------- | --------------------------------------------------------------------- | ------ | --------------- |
| Enum                    | `=` / `≠`                                                             | `=`    | 状态和枚举字段         |
| Boolean                 | `true` / `false`                                                      | `true` | 右值为布尔选项         |
| Number                  | `=` / `≠` / `>` / `≥` / `<` / `≤`                                     | `=`    | 仅节点内置 Number 输出 |
| String                  | `=` / `≠` / `contains` / `not contains` / `is empty` / `is not empty` | `=`    | 文本类字段           |
| Array / Date / DateTime | 不可选                                                                   | —      | 用对应状态/结果变量分岔    |


级联选择与变量生成规则：


| 字段来源            | 选择层级                               | 生成的条件左值                    | 类型     | 运行时取值                             |
| --------------- | ---------------------------------- | -------------------------- | ------ | --------------------------------- |
| Step1 账票 OCR 字段 | `帳票フィールド` → `{帳票タイプ}` → `{OCR字段名}` | `docTypes.{账票类型}.{OCR字段名}` | String | 从上游 OCR 结果聚合                      |
| 数据映射标准字段        | `{映射节点}` → `標準フィールド` → `{标准字段名}`   | `standardFields.{标准字段名}`   | String | 从映射节点 `standardFields` 对象数组按字段名取值 |
| 字段组容器           | `standardFields`                   | 不生成条件左值                    | —      | 仅分组或系统传递                          |


条件变量 catalog：


| 来源           | 变量键                        | 类型     | 取值范围                            | 典型条件场景                                 |
| ------------ | -------------------------- | ------ | ------------------------------- | -------------------------------------- |
| AI 前处理节点     | `preprocessStatus`         | Enum   | `processing / success / failed` | `= failed` → 异常分支                      |
| AI 前处理节点     | `preprocessResult`         | Enum   | `passed / reviewRequired`       | `= reviewRequired` → 前处理确认             |
| OCR 抽出节点     | `ocrStatus`                | Enum   | `processing / success / failed` | `= failed` → 异常分支                      |
| OCR 抽出节点     | `ocrResult`                | Enum   | `passed / reviewRequired`       | `= reviewRequired` → OCR 确认            |
| 数据映射节点       | `mappingStatus`            | Enum   | `processing / success / failed` | `= failed` → 异常分支                      |
| 数据映射节点       | `mappingResult`            | Enum   | `passed / reviewRequired`       | `= reviewRequired` → 人工确认              |
| 数据映射节点       | `standardFields.{field}`   | String | 按 String 运算子                    | 关键字分岔                                  |
| AI 检证节点      | `verifyStatus`             | Enum   | `processing / success / failed` | `= failed` → 异常处理                      |
| AI 检证节点      | `verifyResult`             | Enum   | `passed / reviewRequired`       | `= reviewRequired` → 确认或补件路由           |
| 人工确认节点       | `hitlStatus`               | Enum   | `processing / success / failed` | `= success` → 人工提交完成；`= failed` → 异常分支 |
| 自定义函数节点      | `codeStatus`               | Enum   | `processing / success / failed` | `= failed` → 异常分支                      |
| Step1 账票模板字段 | `docTypes.{账票类型}.{OCR字段名}` | String | 按 String 运算子                    | 字段关键字分岔                                |


条件变量选择器展示约定：

1. 一级显示节点日文类型名 + 节点标识（`varName`）。
2. 二级显示输出变量键原文。
3. 运行时引用路径为 `{节点标识}.{变量键}`。

| 变量选择  | 変数を選択    | 只显示条件变量 catalog |


典型条件：


| 场景              | 条件式                                                                            |
| --------------- | ------------------------------------------------------------------------------ |
| OCR 要确认         | `{ocr}.case.ocrResult = reviewRequired`                                        |
| AI 检证全部通过       | `{verify}.case.verifyStatus = success` 且 `{verify}.case.verifyResult = passed` |
| 映射要确认           | `{map}.case.mappingResult = reviewRequired`                                    |
| 标准字段关键字         | `{map}.case.standardFields.policyNo contains "A"`                              |
| Step1 账票 OCR 字段 | `docTypes.invoice.claimNo contains "A"`                                        |


输出变量：无。

#### 自定义函数节点

节点定位：用户为 Python 脚本添加输入参数，并把每个参数绑定到开始节点 `files[]`、文件类别 JSON 变量或前序节点输出变量。脚本返回值只进执行日志，不进入条件、系统、待办变量池。下游节点只能消费本节点内置 `codeStatus`。

入参规则：

1. 右侧 inspector 的「入力変数」区只展示已添加入参列表；点击「追加」打开入参弹窗。
2. 入参弹窗填写变量名、数据类型、是否必填，并从 Source 下拉选择一个数据来源。
3. 变量名由配置人员自定义，作为 Python 脚本读取键；同一自定义节点内不可重复。
4. 添加完成后，入参列表展示变量名、类型、必填标记和绑定 Source；每条支持编辑和删除。
5. 脚本内用 `inputs['变量名']` 读取已定义输入变量；例如变量名为 `claimFiles` 时，用 `inputs['claimFiles']` 读取。

入参范围：


| 来源类型         | 可选范围                     | 类型处理         | 说明                                     |
| ------------ | ------------------------ | ------------ | -------------------------------------- |
| 开始节点文件数组     | 开始节点 `files[]`           | Array        | 传入案件内文件列表 JSON；用于读取文件名、账票类型、页信息等基础信息   |
| 文件类别 JSON 变量 | Step1 关联账票集合派生的文件类别 JSON | Array / Dict | 可选某一账票类型对应的文件集合或文件对象；用于只处理特定账票         |
| 前序节点输出变量     | 当前自定义节点上游可达节点的输出变量       | 按变量原类型       | 只展示前序可达变量；不可选择后续节点、不可达节点、条件节点输出、结束节点输出 |


入参数据类型：


| 类型       | 用途                    | 示例                                  |
| -------- | --------------------- | ----------------------------------- |
| `string` | 文本、字段值、枚举值            | `inputs['claimNo']`                 |
| `int`    | 整数数量、页码、计数            | `inputs['pageCount']`               |
| `float`  | 金额、分数、置信度             | `inputs['amount']`                  |
| `dict`   | 单个文件对象、单条标准字段对象、结构化结果 | `inputs['mainFile']['fileName']`    |
| `array`  | 文件列表、字段列表、规则结果列表      | `for file in inputs['claimFiles']:` |


配置界面（日语）：


| 区块/控件 | 界面文案（日语）     | 说明                                                 |
| ----- | ------------ | -------------------------------------------------- |
| 面板标题  | カスタム関数       | 右侧检视面板顶栏                                           |
| 区块标题  | 入力変数         | 展示已添加入参；点击「追加」打开入参弹窗；添加后 inspector 只展示绑定结果；支持编辑和删除 |
| 弹窗字段  | 変数名          | 自定义变量名；脚本通过该名称读取；同节点内唯一                            |
| 弹窗字段  | データ型         | `string / int / float / dict / array`              |
| 弹窗字段  | 参照元          | 选择开始节点 `files[]`、文件类别 JSON 变量或前序节点输出变量             |
| 弹窗字段  | 必須           | 开启后运行时若取不到值，节点写 `codeStatus=failed`                |
| 区块标题  | Python スクリプト | 脚本编辑区；脚本通过 `inputs['变量名']` 使用已添加输入变量               |


校验规则：


| 校验项        | 规则                                                              |
| ---------- | --------------------------------------------------------------- |
| 脚本必填       | 空脚本保存阻断                                                         |
| 入参变量名      | 必填；同节点内唯一；只能使用英文字母、数字和下划线，且不能以数字开头                              |
| 入参类型       | 必须从 `string / int / float / dict / array` 中选择                   |
| 入参来源       | Source 必须选择开始节点 `files[]`、文件类别 JSON 变量或前序节点输出变量；不允许选择后续节点或不可达变量 |
| 必填入参       | 运行时必填入参为空、缺失或 Source 不可解析时，节点写 `codeStatus=failed` 并记录执行日志      |
| 脚本输入固定     | 脚本用 `inputs['变量名']` 读取已定义输入变量                                   |
| 脚本返回值不生成变量 | 返回值仅用于本节点日志和调试预览，不进入任何变量池                                       |
| 条件可选限制     | 条件节点只展示本节点内置 `codeStatus`，不展示脚本返回字段                             |
| 失败原因       | 脚本异常、超时、返回格式非法时写执行日志，变量池只写 `codeStatus=failed`                  |


脚本示例：

```python
def main(inputs):
    files = inputs['claimFiles']
    amount = inputs['claimAmount']
    if not files:
        return {'message': 'no files'}
    return {
        'fileCount': len(files),
        'amount': amount,
    }
```

输出变量（配置端不折叠；行内仅变量键+类型；i 只显示变量显示名、取值范围；`files[]` 才显示 1 条短示例）：


| 变量键          | 类型   | 消费分类 | 取值范围                            | 定义                               |
| ------------ | ---- | ---- | ------------------------------- | -------------------------------- |
| `codeStatus` | Enum | 条件   | `processing / success / failed` | Python 脚本处理状态；不输出 result 或错误消息变量 |




#### Step1 账票模板字段

Step1 账票模板字段不是节点输出变量。条件节点可以读取上游 OCR 结果聚合后的账票字段叶子，用于简单分岔。`帳票フィールド` 和 `{帳票タイプ}` 只是级联分组；只有选到第三级 `{OCR字段名}` 后，才按 String 类型展示对应运算子。

条件节点变量级联：


| 级   | 界面文案             | 含义                |
| --- | ---------------- | ----------------- |
| L1  | 帳票フィールド          | Step1 账票 OCR 字段分组 |
| L2  | {帳票タイプ}          | 具体账票类型名           |
| L3  | {OCR Field Name} | 该类型模板中的 OCR 字段    |


变量规则：


| 变量键                        | 类型     | 消费分类 | 可用运算子         | 定义                                |
| -------------------------- | ------ | ---- | ------------- | --------------------------------- |
| `docTypes.{账票类型}.{OCR字段名}` | String | 条件   | 按 String 类型展示 | 从 `files[].ocrFields` 聚合得到的运行时字段值 |

### 6.01 业务场景设置

#### 6.01.1 概述

三步向导：Step1 案件集约规则，Step2 跳转工作流画布，Step3 输出字段与发布。新建、编辑已发布、复制三类场景共用流程；影响执行逻辑的变更将工作流测试结论置为未测试。

#### 6.01.2 功能点

1. 场景列表：新建、复制、重命名、删除、关键词检索；复制后状态为待确认，须复核失效引用。
2. Step1：配置名称、说明、账票集合（1～20）、主账票与主字段、关联账票字段关系、匹配策略；支持关联检查与 AI 自动关联。
3. Step2：进入工作流画布；画布读取 Step1 账票与关系，限制节点可选范围。
4. Step3：勾选并排序输出字段（OCR 字段、标准字段、工作流结果字段等）。
5. 发布：三步校验通过且工作流测试成功后方可发布；新案件使用发布后配置。

#### 6.01.3 页面与入口


| 场景      | 菜单/入口            | 角色                  |
| ------- | ---------------- | ------------------- |
| 场景列表    | 业务规则配置 / 业务场景设置  | 系统管理员、业务规则配置人员、只读人员 |

### 6.02 工作流画布与可编排节点

#### 6.02.1 概述

画布只编排**案件集约完成之后**的处理段；不编排上传、案件集约、导出。节点规则在账票类型、数据映射、AI 检证等设置页维护，画布配置外壳、开关、引用与分支。

- 各节点变量（出力、消费路径、允许取值、可用运算符）见 **6.02.10**。
- 配置端「出力変数」折叠区：变量键 + 类型 + 界面文案；不展示取值说明。文件级统一仅列 `files[]` 一项。

#### 6.02.2 功能点

1. 画布编辑：添加/插入/连接/删除节点与连线、自动整列；逻辑变更置测试为未测试。节点标题固定为类型名。
2. 节点选择器：业务 Agent（前处理、OCR、映射、检证）、控制（条件、人工确认、通知）、扩展（自定义函数）、端点（开始、结束）。
3. 节点编辑面板：按类型展示参数；底部「出力変数」只读。映射、检证仅跳转设置页。
4. 变量引用：条件、通知、函数入参仅选可达上游变量；失效阻断发布。
5. 分支：条件 IF/ELIF/ELSE；人工确认三出口（完成/补件/案件终止），审查者选出口即路由。
6. 人工确认：须接在前处理/OCR/检证之后；只接需要的出口，至少一条。
7. 工作流测试：内置集约済み用例；不跑上传/集约，不写正式案件/待办。

#### 6.02.10 工作流变量

本节含跨节点共用约定与各节点变量一览；各节点小节统一结构：节点说明 → 配置界面（日语）→ 画布摘要（日语）→ 变量一览（含**消费路径**、允许取值、可用运算符、定义）。

**画布摘要通则**：节点标题下方单行展示；段间以 `·` 分隔；无有效配置时显示 `未設定`（数据映射无已配规则时显示 `設定参照`）；开始/结束节点不显示摘要。结构上首段为计数或状态摘要（如 `{n}/{total} 有効`、`{n}件有効`、`ルール {n}件`），后续段为明细项（能力名、模块名、对象名称等），明细最多再展示 2 项。

##### 6.02.10.1 作用域与变量消费路径

**消费路径**（变量表「消费路径」列）：标明该变量供哪类配置/界面使用。**条件**与**待办·通知**不要混用——分岔看聚合态，明细给人看。


| 消费路径      | 配置入口                  | 用途                                            | 典型变量                                                                                                                                  |
| --------- | --------------------- | --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| **条件**    | 条件节点「変数を選択」           | 画 IF/ELIF 分岔：判断「要不要走人工/补件/异常」                 | `*Status`、`verifyStatus`、`preprocessStatus`、`lowConfidenceFieldCount`、`caseStatus`、`hasOpenItems`、`standardFields` 叶子、Step1 账票 OCR 字段 |
| **待办**    | 无手动配置；引擎写入 open 待办时带入 | 人工确认界面展示：要审什么文件、缺什么、冲突在哪                      | `files[]`（含 `ocrFields`/`failedRules`/`manualEdits`）、`mappingConflicts`、`mappingErrors`、`aiVerifyResultJson`                          |
| **通知**    | 通知节点「変数を挿入」           | 件名/正文模板：`{{caseNo}}`、`{{missingDocuments}}` 等 | `caseNo`、`missingDocuments`、`missingFields`、`errorMessage`、`endedAt`、`lastFailureReason`                                              |
| **待办·通知** | 待办展示 + 通知可插入          | 同一份明细既给审查员看，也进通知文案                            | `missingDocuments`、`missingFields`                                                                                                    |
| **运行时**   | 三类 UI 均不选             | 引擎内部、Step3 导出、执行日志、状态机汇聚                      | `caseId`、`branchName`、`docTypes[]`、`businessScene`                                                                                    |


**同一业务事件的三层写法**（以必要书类不足为例）：


| 层次   | 变量                       | 示例                       |
| ---- | ------------------------ | ------------------------ |
| 条件分岔 | `requiredDocumentStatus` | `= missing` → 接人工确认或补件分支 |
| 待办明细 | `missingDocuments`       | 待办界面列表：`['診断書','領収書']`   |
| 通知正文 | `missingDocuments`       | 「不足書類：診断書、領収書」           |


**作用域**


| 作用域   | 路径格式                 | 出力変数面板          | 条件左值                        |
| ----- | -------------------- | --------------- | --------------------------- |
| 案件    | `{节点}.case.{字段}`     | 列出 case 字段      | 仅「消费路径」含 **条件** 的字段         |
| 文件    | `{节点}.files[]`       | 仅列 `files[]` 一项 | 否；供 **待办** 展示               |
| 账票类型级 | `{start}.docTypes[]` | 仅开始节点           | 否（整对象）；Step1 字段见 6.02.10.13 |


通用规则：

- 仅选**上游连线可达**变量；失效阻断发布。
- `files[]` 及 `files[].*` **不可**作条件左值；消费路径为 **待办**（人工确认看原图/字段/违规规则）。元素结构见 6.02.10.2。
- **案件级 Array 明细**（`missingDocuments`、`missingFields`、`mappingConflicts`、`mappingErrors`）消费路径为 **待办·通知** 或 **待办**，**不出现在条件选择器**；分岔用对应 `*Status` / `verifyStatus` / `mappingStatus`。
- **标准字段**：映射节点产出，键见总 PRD 6.04；条件须选叶子 `{map}.case.standardFields.{变量名}`（中间层「標準フィールド」），整对象不可选。
- **Step1 账票字段**：非节点输出；条件级联须选至 OCR 字段叶子。界面三级路径：`帳票フィールド` → `{帳票タイプ}` → `{OCRフィールド名}`；运行时路径 `docTypes.{帳票タイプ}.{OCR字段名}`。详见 6.02.10.13。
- 控制节点（条件/通知/结束）无 `files[]`。
- **不作条件左值**（仍可在出力変数面板列出）：`caseId`、`caseNo`（仅通知插入）、`businessScene`、`triggerType`、`docTypes[]` 整对象；Workflow 已移除的节点类型（マスタ照合、MCP）亦无变量规格。
- 各变量**允许取值**与**可用运算符**见对应节点变量一览表。
- **条件组合**：同一 IF/ELIF 分支内多行条件以 **AND / OR** 药丸连接（见 6.02.10.10）。多值匹配（如「状态为補件或人工確認」「数组同时含 A 与 B」）通过多行条件 + OR/AND 表达，**不提供** `いずれか` / `いずれかを含む` / `すべて含む` 等聚合运算符。
- **运算符分层**（变量表仅列该层推荐项；节点必写 Enum/Boolean 不用 `未設定`/`設定済み`）：


| 类型               | 可用运算符                                                         | 默认      |
| ---------------- | ------------------------------------------------------------- | ------- |
| Enum（状态）         | `=` / `≠`                                                     | `=`     |
| Boolean          | `true` / `false`                                              | `true`  |
| Number           | `=` / `≠` / `>` / `≥` / `<` / `≤` / `範囲内`                     | `=`     |
| String（必有值）      | `=` / `≠` / `含む` / `で始まる`                                     | `=`     |
| String（可选/失败时写入） | 上列 + `未設定` / `設定済み`                                           | `含む`    |
| Array            | `件数 =/>/≥/</≤` / `含む`                                         | `件数 >`  |
| DateTime         | `=` / `≠` / `>` / `≥` / `<` / `≤` / `範囲内` / `直近N日以内` / `N日以前` | `=`     |
| Object           | `設定済み` / `未設定` / `キーを含む`                                      | `設定済み`  |
| 标准字段叶子           | 按 6.04 字段 dataType 套用上行                                       | 同左值类型默认 |


**人工分岔**：由条件节点 → 人工确认节点决定；常用变量见各 Agent 节点小节。

##### 6.02.10.4 结束节点

节点说明：

分支路径终态锚点；到达时写入变量，参与案件状态汇聚与 Step3 导出。不接收触发事件。

未完了事項（业务概念）与 hasOpenItems（布尔输出）：

- 未完了事項：分支到达结束节点时，仍阻塞本案在本分支配下业务办结的事项集合。涵盖 open 人工待办、待确认文件/字段、补件待上传、跨批次 pending 未汇总、成果文件未生成、处理未完结的逻辑文件等。是业务语义；不单独作为输出变量键暴露。
- hasOpenItems（未完了事項あり）：结束节点写入的布尔快照，表示上述集合在写入时刻是否非空。true=仍有未完了事項；false=均已消除或不适用。参与案件状态汇聚、条件分岔与 Step3 导出。
- 计算时机：结束节点执行完成时，引擎扫描全案快照一次性判定并写入；与 caseStatus、finalResult、resultFileStatus 同期产出。
- 纳入范围（任一命中即 hasOpenItems=true）：open 人工待办（created 状态的前处理/OCR/检证待办，或缓冲等待中尚未触发的待确认项）；待确认文件/字段（仍待人工确认的文件，或未消除的低置信/分类待确认 OCR 字段）；补件待上传（finalResult=補件待ち，或必要书类/字段仍 missing 且尚未收到并归属补件文件）；跨批次 pending 未汇总（跨批次上传或补件绑定产生的 pending 项尚未汇总完成）；成果文件未生成（Step3 已配置导出且 resultFileStatus 为 待機 / 生成中 / 失敗）；处理未完结的文件（files[] 中存在 Pending / Processing / Failed 状态）。
- 排除：已关闭待办、已汇总 pending、skipped 检证模块、resultFileStatus=対象外（未配置 Step3 导出）。
- 判定示例：正常办结且无 open 待办、成果文件已生成 → false；finalResult=補件待ち → true；到达结束但 OCR 确认待办仍为 open → true；Step3 导出失败 → true；跨批次 pending 缓冲未结束 → true。

##### 6.02.10.10 条件判断节点

节点说明：

IF / ELIF 每行一条比较式；同分支多条件以 **AND / OR** 药丸连接（默认 AND）；ELSE 无表达式。多值匹配不写聚合运算符，用多行 + OR/AND 表达。左值从上游可达 catalog 选取（规则见 6.02.10.1）。

**纳入规则**：可达上游处理节点；排除 `files[]`、整对象 `standardFields`/`docTypes[]`、案件级 Array 明细（`missingDocuments` 等）；标准字段选叶子（中间层「標準フィールド」）；Step1 账票字段须选三级路径叶子，见 6.02.10.13。

**典型示例**：


| 场景              | 条件式                                                                            |
| --------------- | ------------------------------------------------------------------------------ |
| OCR 低置信 → 人工确认  | `{ocr}.case.lowConfidenceFieldCount` `>` `0`                                   |
| 必要書類不足          | `{verify}.case.requiredDocumentStatus` `=` `missing`                           |
| 补件态             | `{start}.case.caseStatus` `=` `補件`                                             |
| 多状态（OR）         | `{start}.case.caseStatus` `=` `補件` **OR** `{start}.case.caseStatus` `=` `人工確認` |
| 映射金额            | `{map}.case.standardFields.claimAmount` `>` `0`                                |
| Step1 账票 OCR 字段 | `docTypes.{帳票タイプ}.{OCR字段名}` `含む` `{比较值}`                                       |
| 检证全过            | `{verify}.case.verifyStatus` `=` `success`                                     |


各节点可选变量及其允许取值、可用运算符见对应节点变量一览表。

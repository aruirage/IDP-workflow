# 跨批次案件并入、自动补件与人工触点去重 — 设计说明

> 讨论定稿：2026-07-09  
> 关联 PRD：`PRD-idp-workflow-kisoku.md`

## 1. 背景与问题

用户从**主上传入口**分批上传同一申请的材料（时间相近、案件识别字段相同、账票类别可能不同）。若每批独立处理，会出现：

1. 第二批次错误地**新建第二个案件**；
2. 同一案件内，多批次文件分别触发**多条人工确认待办**；
3. **通知重复发送**（同一节点、同一案件）。

补件**独立上传入口**保留为兜底；主路径改为**统一主上传 + 双池自动路由**。

---

## 2. 设计原则

| 原则 | 说明 |
|------|------|
| 统一主上传 | 主上传为默认入口；系统按路由阶段与识别字段自动分流 |
| 双池检索 | 先 **Open Case 池**（并入），再 **补件候选池**（自动补件） |
| 案件在第一批新建 | 批次 1 案件集约成功 → **新建案件 A**（路由阶段 `pre_ai_verify`） |
| 第二批只并入 | 批次 2 命中 Open Case 池 → **并入 A，不新建** |
| 已到 AI 检证自动补件 | 命中补件候选池 → **自动绑定补件**，不新建，不重新集约 |
| 匹配在案件集约/路由 | 复用 Step1 规则（精准/近似、字段标准化） |
| 自动节点并行 | 新并入/补件文件独立跑前处理/OCR，不重跑已有文件 |
| 人工触点去重 | 同一案件、同一节点类型 → **一条待办 / 一次通知** |
| 人工触点策略 | 待办合并系统默认；缓冲为**场景级**可选（前处理/OCR）；AI 检证入口统一等待 |

---

## 3. 路由阶段与双池

### 3.1 路由阶段（系统内部）

| 路由阶段 | 含义 | 双池 |
|----------|------|------|
| `pre_ai_verify` | AI 检证前 | Open Case 池 |
| `post_ai_verify` | AI 检证及之后 | 补件候选池 |
| `closed` | 处理完成 / 处理中止 | 不参与检索 |

转换：新建/并入 → `pre_ai_verify`；首次到达 AI 检证 → `post_ai_verify`；结束 → `closed`。

### 3.2 双池检索顺序

主上传完成分割后，案件集约/路由阶段：

```
1. 查 Open Case 池（pre_ai_verify + 未终结 + 唯一命中 → 并入）
2. 若未命中，查补件候选池（post_ai_verify + 未终结 + 唯一命中 → 自动补件）
3. 0 命中 → 新建
4. 多命中 / 低置信 → 人工确认（集约确认或补件归属确认）
```

### 3.3 Open Case 池（跨批次并入）

| 项 | 规则 |
|----|------|
| 检索对象 | 同业务场景 + `pre_ai_verify` + 未终结案件 |
| 匹配规则 | Step1 主账票驱动、案件识别字段、字段关系、精准/近似、字段标准化 |
| 本批次有主账票 | 识别字段唯一命中 open case → 并入 |
| 本批次仅有关联文件 | 关联文件字段 ↔ 已有案件主账票字段反向匹配 |
| 唯一命中 | 归入已有案件，**不新建** |
| 0 命中 | 新建案件 |
| 多命中 / 低置信 | 人工集约确认 |

### 3.4 补件候选池（自动补件绑定）

| 项 | 规则 |
|----|------|
| 检索对象 | 同业务场景 + `post_ai_verify` + 未终结案件 |
| 唯一命中 | 自动绑定补件，**不新建**，**不重新集约** |
| 处理范围 | 从前处理重跑到 AI 检证 |
| 兜底 | 案件详情补件入口（多候选、自动绑定失败等） |

> 已废弃「智能分流提示跳转补件」作为主路径；自动补件为默认行为，补件入口为兜底。

## 4. Workflow：新文件处理

案件 A 并入或自动补件绑定后：

| 项 | 规则 |
|----|------|
| workflow 实例 | 始终**一个**（案件 A） |
| 前处理 / OCR / Data Mapping | **仅对新文件**执行，不重跑已有文件 |
| 未分类文件 | **不参与** workflow |
| 节点输出 | 按文件/实例归档；案件级变量随新文件更新 |

### 4.1 未分类剔除（文件分割阶段）

文件分割沿用其他产品线已有规则，本设计不重复定义。与本 Workflow 相关的边界结果如下：

| 项 | 规则 |
|----|------|
| 判定 | 分割结果**未命中**本场景 Step1 已选账票类型 |
| 标记 | **未分类** |
| 处理 | **不进入**归集与 Workflow |
| 展示 | 分割结果中保留，供业务查看 |

## 5. 人工确认待办：合并只发一次

### 5.1 去重键

```
去重键 = 案件 ID + 工作流节点 ID + 人工确认上下文类型
```

同一去重键在任意时刻**最多存在一条 open 待办**。

### 5.2 触发规则

| 情况 | 行为 |
|------|------|
| 无 open 待办 | 创建一条待办 |
| 已有 open 待办 | 追加新文件/字段，不新建 |
| 待办已提交，后续又有新文件 | 新建待办，仅含未确认新增项 |

适用：前处理 / OCR 识别 / AI 检证 人工确认。

### 5.3 人工确认前缓冲（场景级，可选）

| 项 | 规则 |
|----|------|
| 配置位置 | **Step2 场景级「人工触点策略」**（不在 HITL 节点内配置） |
| 开关 | 缓冲等待 + 时长（5/15/30/60 分钟）；适用 **前处理确认、OCR 确认** |
| 缓冲行为 | 需人工确认时先缓冲；新文件就绪则**滑动延长** |
| 缓冲结束 | 合并 pending 项，只触发一条待办 |
| 用户可见 | 案件状态仍为**处理中**；`待处理事项.等待人工缓冲 = 是` |

AI 检证确认：系统默认（AI 检证入口等待 + open 去重），客户不可配。详见 PRD §1.3、§1.4。

---

## 5A. AI 检证入口（跨批次 / 补件统一）

跨批次并入与补件绑定/续跑 **共用同一套 AI 检证入口事件**；差异仅在于触发来源与路由阶段（PRD §1.2）。

| 入口事件 | 说明 |
|----------|------|
| 本批新文件处理完成 | 前处理 / OCR / Mapping |
| 上游人工待办已关闭 | 本批若经 HITL |
| pending 汇总完成 | 跨批次或补件 pending |

**AI 检证输入** = 整案快照（旧 + 新文件），合并检证一次。前处理/OCR/Mapping 仍只跑新文件。

缺件 → 等待补件 → 补件到达 → 续跑 → 同一入口 → 整案再检（PRD §1.3）。

---

## 6. 通知：合并只发一次

```
去重键 = 案件 ID + 工作流节点 ID（通知节点）
```

同一去重键在同一 workflow 执行周期内**只发送一次**。

---

## 7. 案件状态双轨模型

| 层 | 内容 |
|----|------|
| 用户可见 | 处理中、人工确认待处理、等待补件、异常处理、处理完成、处理中止 |
| 系统内部 | 路由阶段 `pre_ai_verify` / `post_ai_verify` / `closed` |

Open Case **不是**单独用户状态，由 `pre_ai_verify` 驱动。

## 8. 测试用例建议

| # | 场景 | 预期 |
|---|------|------|
| 1 | 批次 1 新建 A，批次 2 同字段（pre 池） | 并入 A，不新建 B |
| 2 | 批次 2 仅关联文件，反向匹配 A | 并入 A |
| 3 | 批次 1 OCR 待办 open，批次 2 也需 OCR 确认 | **一条** OCR 待办 |
| 4 | OCR 待办已提交，批次 2 新文件需 OCR 确认 | **一条新**待办，仅含批次 2 |
| 5 | 两批文件先后到达通知节点 | **只发一次**通知 |
| 6 | A 在 post 池，批次 2 主上传同字段 | **自动补件绑定**，不新建 |
| 7 | 同时 pre/post 两池可命中 | **先** Open Case 并入 |
| 8 | OCR 节点开启 30 分钟等待，两批先后需确认 | 缓冲合并，一条待办；列表显示处理中 + 子标志 |
| 9 | 识别字段多候选 | 人工确认，不自动并入/补件 |
| 10 | 扫描含广告页 | 未分类剔除，有效页正常归集 |
